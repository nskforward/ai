# Архитектура AI-Агента (Go Framework)

Данный документ описывает устройство и жизненный цикл универсального AI-агента, разработанного на базе этого фреймворка. Архитектура построена с акцентом на автономность, расширяемость (cloud/local) и безопасность.

## 1. Основные компоненты

### Драйвер ввода-вывода (Transport)
Агент не привязан к конкретному источнику данных. Взаимодействие абстрагировано через интерфейс:
```go
type Message struct {
    SessionID string // В рамках какой сессии идёт диалог
    UserID    string // Идентификатор пользователя для авторизации команд
    Text      string // Текст запроса
}

type Transport interface {
    // Блокирующий метод, ожидающий новое пользовательское сообщение
    Read() (Message, error)
    // Метод отправки ответа пользователю
    Write(sessionID string, modelOutput string) error
}
```
Фреймворк из коробки предоставляет `ConsoleTransport` (где `UserID = "admin"` и `SessionID = "cli"`), чтобы сразу тестировать агента без написания ботов в Telegram.

### Хранилище памяти (Storage)
Все данные (системный промпт, опыт, скиллы) представлены в виде файлов (документов). Чтобы в будущем можно было переключиться с локальных файлов на S3 или базу данных, введена абстракция:
```go
type Storage interface {
    Read(path string) ([]byte, error)
    Write(path string, data []byte) error
    List(dir string) ([]string, error)
}
```
По умолчанию используется локальная файловая система `storage.LocalFS`.

### Системный промпт и Авто-создание
Системный промпт не "захардкожен". При запуске фреймворка, геттер промпта ищет в `Storage` файл с системным промптом (например, `sysprompt.md`). Если файл отсутствует, агент создает его с базовым (дефолтным) наполнением.
Любые ручные изменения администратором в `sysprompt.md` на диске будут подхвачены агентом "на лету" без остановки приложения.

### Две модели (Light & Heavy Models)
- **Light Model**: Обрабатывает простое сообщение, сверяется с "Оглавлением", решает мелкие задачи.
- **Heavy Model**: Вступает в дело для сложных рассуждений и написания длинных инструкций опыта.

## 2. Память и Обучение (Skills)

1. **Оглавление**: Фреймворк сканирует `Storage` (папка `skills/`) и генерирует "Оглавление" накопленного опыта.
2. Это Оглавление динамически вшивается в Системный Промпт.
3. Light Model видит нужный файл, читает его через Tool и может делегировать Тяжелой модели вместе с готовой инструкцией.
4. В конце сложной задачи, агент пишет через инструмент (Tool) новый шаг в `Storage`.

## 3. Авторизация и Безопасность

### Доступ пользователей (Admins)
Не каждый пользователь (UserID из Transport) может просить агента изменить `sysprompt.md` или записать/удалить навыки:
- В конфигурации агента задается белый список `AllowedAdmins []string`.
- Инструменты (Tools), модифицирующие системную память (UpdateSystemPrompt, MemoryUpdate), скрыты или выдают ошибку `Permission Denied`, если запрос поступил от неавторизованного `UserID`.

### Песочница (Sandbox)
Для предотвращения деструктивных действий от LLM, все базовые функции для работы с файлами и сетью обернуты:
- **Storage Sandbox**: Запрещен выход из каталога (`../`).
- **Network Sandbox**: Белый список разрешенных доменов для HTTP.
