# Анализ архитектуры и предложения по улучшению

## Общие замечания

Архитектура в целом хорошо структурирована и следует принципам чистой архитектуры. Однако есть несколько областей для улучшения:

---

## 1. **User — отсутствует в основной архитектуре Agent**

В документе определён интерфейс `User`, но он **нигде не используется** в `Agent` или `Transport`. 
- В `Transport.Message` есть `Session`, но непонятно, как `User` связывается с `Session`.
- Рекомендуется добавить `User` в `transport.Message` или `Session`.

---

## 2. **Transport.Message — конфликт имён с LLM.Message**

Оба определения называются `Message`, что может вызывать путаницу при использовании:
```go
type Message struct {
    Session Session
    Text    string
}
```

**Рекомендация:** Переименовать в `TransportMessage` или добавить пакетный импорт с алиасом.

---

## 3. **Transport.SendEvent — сигнатура некорректна**

```go
SendEvent(session, Event) error
```

Первый аргумент должен быть типизирован: `session Session` (а не просто `session`).

---

## 4. **LLM.Message — отсутствует поле Name для ToolCalls**

В `ToolCall` есть поле `Name`, но в `Message` нет аналогичного для роли `Tool`. Также хорошо было бы добавить:
- `Name` — для именованных сообщений
- `Timestamp` — для отладки и аудита

---

## 5. **Tool.Allow — зависимость от нетипизированного session.Session**

```go
Allow(session.Session) error
```

Лучше определить отдельный интерфейс `Session` или передавать `User`. Сейчас непонятно, какие методы должны быть у `Session`.

---

## 6. **Storage определён, но не используется**

`FileStorage` описан, но не подключён к `Agent.Config`. Если это планируется для tool — хорошо. Но тогда `Tool` должен иметь доступ к `Storage` через конструктор или контекст.

---

## 7. **Agent.Config — проблемы с дизайном**

- `Tools []tool.Tool` — лучше использовать `ToolRegistry` напрямую, а не слайс
- `AllowedAdmins` лучше вынести в отдельный интерфейс `Authenticator` или `PermissionChecker`
- Нет поля для `SessionStore` в Config — он создаётся отдельно и хранится в Agent

---

## 8. **SessionStore — нет метода для получения всех sessionID**

Для мониторинга или очистки stale сессий полезно иметь:
```go
GetAllIDs() []string
```

---

## 9. **Middleware — отсутствует тип для ошибок**

Рекомендуется определить свои ошибки (например, через `errors` пакет) для:
- `ErrForbidden`
- `ErrUnauthorized`
- `ErrRateLimited`

---

## 10. **Event — слишком ограничено**

```go
type Event uint8
const InProcessing Event = 1
```

Хорошо бы расширить или сделать более гибким (например, через string или интерфейс).

---

## 11. **Отсутствует описание потока данных**

Было бы полезно добавить диаграмму или описание того, как сообщение проходит от Transport через Middleware к LLM и Tools.

---

## Итог

Архитектура требует доработки в части:
- Интеграции `User` в поток данных
- Устранения конфликтов имён
- Более строгой типизации
- Расширения функциональности `SessionStore` и `ToolRegistry`
